# Завдання 4. Розглянути, чи безпечно створювався додаток

---

## Оцінка безпеки

| Принцип безпеки | Вимога (за матеріалом) | Оцінка |
|-----------------|------------------------|---------|
| **Режим налагодження (DEBUG)** | Має бути встановлений у `False` у продакшні, щоб припинити виведення налагоджувальної інформації. | Виконано, якщо підготовка до продакшну була правильною. |
| **Секретний ключ (SECRET_KEY)** | Не повинен вказуватися у вихідному коді. Рекомендується розміщувати у змінній оточення або файлі з обмеженим доступом. | Виконано, якщо ключ не захардкоджено. |
| **Захист від XSS** | Використовувати вбудований захист шаблонів Django. Не використовувати `safe` або `mark_safe` для контенту від користувача. | Ймовірно виконано, якщо використано стандартні шаблони. |
| **Захист від CSRF** | Не вимикати захист (`csrf_exempt`). Використовувати `SESSION_COOKIE_SECURE` та `CSRF_COOKIE_SECURE` при HTTPS. | Ймовірно виконано, якщо дотримувалися правил. |
| **Перевірка заголовка Host** | Параметр `ALLOWED_HOSTS` має бути заповнений конкретними доменами. | Виконано, якщо налаштовано під час деплою. |
| **Завантаження файлів (Media)** | Медіафайли мають роздаватися з окремого домену, щоб уникнути ризиків пов'язаних із **same-origin policy**. | Невідомо / потенційно вразливо, якщо все на одному домені. |

---

## Висновок

Застосунок, створений на уроці, **безпечний на рівні коду**, оскільки Django надає вбудований захист від основних атак:

- XSS  
- CSRF  
- SQL Injection  

Однак **безпека в продакшні залежить від налаштувань деплою**:

- `DEBUG=False`
- прихований `SECRET_KEY`
- налаштований HTTPS / SSL / HSTS
- правильно заданий `ALLOWED_HOSTS`
- безпечне розповсюдження медіафайлів

Тому повна безпека залежить від того, чи всі інфраструктурні кроки були виконані коректно.
